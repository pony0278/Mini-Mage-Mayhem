# 魔法事故報告：收容測試 設計整理

版本：v0.1 概念整理  
用途：重新定位「魔法大亂鬥」的核心缺口，強化社群分享性、戲劇性與玩法記憶點。

---

## 1. 核心發現

原本的「魔法大亂鬥」雖然已經有不錯的手感、法術組合與短局戰鬥，但缺少一個很強的社群傳播點。

目前缺少的不是更多怪物、更多法術或更多關卡，而是：

> 玩家每一局玩完後，有沒有一件「很想截圖、分享、講給朋友聽」的事情？

單純打怪比較容易變成：

> 我進場、打怪、升級、通關或死亡。

但真正有分享性的遊戲，通常會讓玩家產生：

> 這一場也太荒謬了，我要給朋友看。

因此，「魔法事故報告」不是單純的結算畫面，而是可以成為整個遊戲的新核心。

---

## 2. 新核心定位

### 原本定位

> 魔法師使用不同法術，在場地中亂鬥、打怪、打 Boss。

### 新定位

> 在一座極度不安全的魔法實驗基地中，玩家扮演失控法師或收容員，互相追捕、反抗、製造混亂。每一局結束後，都會產生一份荒謬的「魔法事故報告」。

### 一句話版本

> 這不是一款單純打怪的魔法遊戲，而是一款每場都會產生離譜事故的魔法收容亂鬥。

### 更適合平台介紹的版本

> 抓住失控法師，把他關進實驗艙；或扮演失控法師，逃出基地、反炸收容員，製造一份最荒謬的魔法事故報告。

---

## 3. 世界觀包裝

遊戲背景可以設定在一個「魔法事故管理局」或「魔法學院安全委員會」底下。

玩家不是傳統英雄，而是基地中的高風險魔法人員。

### 主要角色身份

| 身份 | 說明 |
|---|---|
| 失控法師 / 受測體 | 法術能量不穩定，被判定需要收容或校正。 |
| 收容員 / 實驗員 | 負責把失控法師抓回安全艙。 |
| 安全委員會 | 結算時負責產生事故報告與吐槽評語。 |
| 魔法基地 | 充滿實驗艙、導電地板、爆炸核心、冷卻水池與不安全機關的場地。 |

### 語氣方向

建議走 Q 版、荒謬、搞笑，而不是恐怖或黑暗人體實驗。

「抓人關進實驗艙」如果做得太陰暗，會變得像殘酷實驗；但如果改成「安全校正艙」、「魔力冷卻艙」、「反省泡泡艙」，就會變得比較適合 Poki / CrazyGames 這類輕玩家平台。

## 4. 核心玩法：收容測試

## 模式名稱

> 魔法事故報告：收容測試

或：

> Magic Incident Report: Containment Test

---

## 5. 一局基本規則

### 玩家配置

初期可以先做 1v1，之後再擴展到 1v2、1v3 或 2v2。

| 陣營 | 人數 | 目標 |
|---|---:|---|
| 失控法師 | 1 | 撐過時間、逃出基地、破壞實驗艙。 |
| 收容員 | 1～3 | 削弱失控法師，將他推入或拖入實驗艙完成收容。 |

---

## 6. 收容員的玩法流程

收容員不是單純把對方打死，而是要完成三段式收容。

### 第一階段：削弱魔力

收容員需要透過攻擊、法術、陷阱或場地機關，降低失控法師的「穩定值」。

可能手段：

- 冰法術讓失控法師變慢。
- 雷法術讓失控法師短暫麻痺。
- 土法術製造障礙或困住對方。
- 風法術把對方推向實驗艙。
- 魔法鎖、捕捉網、泡泡陷阱短暫控制對方。

### 第二階段：推進或拖往實驗艙

失控法師被削弱後，收容員可以嘗試把他推進實驗艙。

這裡是主要笑料來源：

- 收容員推人時被反炸。
- 地板結冰，雙方一起滑走。
- 失控法師掙脫後反過來把收容員推進艙裡。
- 收容員打錯方向，把隊友推飛。
- 實驗艙門被雷電短路，導致收容失敗。

### 第三階段：關艙倒數

把失控法師推進艙後，不是立刻勝利，而是開始 3 秒倒數。

倒數期間：

- 收容員要守住艙門。
- 失控法師可以掙脫或用最後法術干擾。
- 場地機關可能打斷收容。
- 如果失敗，實驗艙可能過載爆炸，把所有人炸飛。

成功後：

> 收容完成。

失敗後：

> 收容艙過載，事故升級。

---

## 7. 失控法師的玩法

失控法師不是單純逃跑，而是要一路製造混亂。

可能能力：

| 能力 | 用途 |
|---|---|
| 隨機元素爆發 | 造成不可預期的混亂。 |
| 陷阱反轉 | 把收容員設下的陷阱變成自己的武器。 |
| 破壞實驗艙 | 暫時讓收容點失效。 |
| 偷走法術 | 短暫使用收容員的能力。 |
| 引爆場地物件 | 製造爆炸、遮蔽或逃生空間。 |

失控法師的勝利條件可以是：

- 撐過 2 分鐘。
- 逃到出口。
- 成功破壞主要實驗艙。
- 讓收容員發生重大事故。

## 8. 場地設計：魔法實驗基地

場地不應只是普通競技場，而應該是「會出事的基地」。

### 場地房間範例

| 房間 | 功能 |
|---|---|
| 實驗艙室 | 收容員要把失控法師推進去的主要目標區。 |
| 元素反應室 | 火、冰、雷、風容易產生連鎖反應。 |
| 冷卻水池 | 可滅火、導電、滑倒或觸發蒸氣爆炸。 |
| 雷電導線區 | 雷系法術會沿著導線擴散。 |
| 魔法倉庫 | 有爆炸桶、魔法核心、可推動物件。 |
| 失控核心房 | 高風險高回報區，容易造成大爆炸。 |
| 傳送門走廊 | 失控法師可逃跑，收容員也可能追錯方向。 |
| 觀察室 | 世界觀裝飾，可放置安全委員會告示。 |

### 場地原則

場地的目標不是公平空曠，而是製造「可理解的混亂」。

好的場地應該讓玩家清楚知道：

- 這裡很危險。
- 但我可以利用危險。
- 如果失控，會很好笑。

## 9. 元素事故系統

原本的法術組合可以轉化為事故來源。

| 法術組合 | 正常用途 | 事故效果 |
|---|---|---|
| 火 + 冰 | 製造蒸氣爆破 | 近距離會把自己或隊友炸飛。 |
| 雷 + 水 / 蒸氣 | 大範圍導電 | 可能連到自己或讓艙門短路。 |
| 風 + 火 | 火焰龍捲 | 可能被牆面反彈，追著玩家跑。 |
| 土 + 雷 | 磁暴震擊 | 可能吸引金屬物件或陷阱。 |
| 冰 + 風 | 暴風雪 | 地板結冰，自己也可能滑倒。 |
| 火 + 土 | 熔岩裂縫 | 可能阻斷自己的逃生路線。 |

### 設計重點

法術不只是強弱問題，而是：

> 越強的法術，越可能造成失控；但高手可以利用失控。

這會讓遊戲比一般技能對打更有記憶點。

## 10. 魔法事故報告：結算系統

每一局結束後，系統根據玩家行為產生一份事故報告。

這是遊戲的社群分享核心。

### 報告內容可以包含

- 事故編號
- 事故名稱
- 事故等級
- 受測體名稱
- 收容員名稱
- 地點
- 使用法術
- 事故摘要
- 勝負結果
- 基地損害率
- 誤傷次數
- 自傷次數
- 最高連擊
- 收容成功 / 失敗原因
- 安全委員會評語
- 挑戰碼 / 分享碼

---

## 11. 事故報告範例

```text
魔法事故報告 #104

事故名稱：收容艙反向吸入事件
事故等級：S

受測體：雷火失控法師
收容員：2 名
地點：第三元素反應實驗室

事故摘要：
收容員成功將受測體推入實驗艙。
但由於艙門在關閉前遭雷鏈短路，
收容程序反向啟動，導致一名收容員被吸入艙內。
受測體趁亂逃脫，並額外炸毀 4 個冷卻裝置。

結果：
收容失敗
基地損害：87%
誤傷隊友：3 次
受測體逃脫時間：02:11

安全委員會評語：
「技術上來說，有人被成功收容了。只是收錯人。」
```

## 12. 事故等級

事故等級不只是評價勝負，而是評價「這一場有多荒謬」。

| 等級 | 名稱 | 條件範例 |
|---|---|---|
| E | 輕微失誤 | 普通失敗，沒有特殊連鎖。 |
| D | 小型事故 | 有自傷、滑倒、撞牆等小事件。 |
| C | 場地損壞 | 炸毀大量物件或破壞機關。 |
| B | 危險收容 | 觸發元素連鎖、多人被波及。 |
| A | 嚴重事故 | 收容過程失敗，造成基地重大損害。 |
| S | 災難級事故 | 收容員、受測體、場地一起失控。 |
| S+ | 禁術級事故 | 以極度離譜的方式完成勝利或全滅。 |

---

## 13. 事故名稱與觸發條件

| 事故名稱 | 觸發條件 |
|---|---|
| 反向收容事件 | 收容員被關進實驗艙。 |
| 受測體自行入艙事件 | 失控法師因滑倒、擊飛或操作失誤自己進艙。 |
| 艙門短路事件 | 關艙倒數被雷系法術打斷。 |
| 三方全責爆炸 | 所有人同時被同一場爆炸波及。 |
| 過度執法事件 | 收容員造成的基地損害比受測體更高。 |
| 錯誤目標收容 | 抓到隊友或 NPC。 |
| 冷卻系統失效 | 火系法術引發基地過熱或蒸氣爆炸。 |
| 安全規範不存在事件 | 同一局觸發 5 種以上事故。 |
| 冰面送貨事件 | 失控法師被冰面一路滑進實驗艙。 |
| 自爆協助收容事件 | 失控法師自己的爆炸把自己炸進艙。 |
| 友軍推進事故 | 收容員把隊友推進危險區或實驗艙。 |
| 實驗艙過載事件 | 關艙失敗後艙體爆炸。 |

## 14. 稱號系統

每局可以給玩家一個稱號，提高分享性。

| 玩家行為 | 稱號 |
|---|---|
| 一直使用火 + 冰 | 蒸氣爆破法師 |
| 自傷很多但仍然勝利 | 自爆倖存者 |
| 用雷鏈連鎖多人 | 導電災難源 |
| 靠風把人推進艙 | 風壓收容員 |
| 被自己法術害死 | 自作自受研究員 |
| 收容員被關進艙 | 反向收容受害者 |
| 大量誤傷隊友 | 團隊安全威脅 |
| 逃跑者反抓收容員 | 受測體升格主管 |
| 一路滑進艙 | 冰面快遞包裹 |
| 破壞基地但完成任務 | 合格但不可取 |

## 15. 社群分享設計

這個方向最重要的目標是：

> 每一局都可以產生一個值得分享的故事。

### 分享形式

初期不需要伺服器，也可以做：

1. 結算分享文字。
2. 事故報告圖片卡。
3. 挑戰碼。
4. 每日事故重現。
5. 本機最佳事故紀錄。

### 分享文字範例

```text
我在《魔法事故報告》觸發了 S 級事故：收容艙反向吸入事件。
原本要抓失控法師，結果收容員自己被關進實驗艙。
基地損害 87%，安全委員會評語：「至少有人被收容了。」
挑戰碼：MIR-7A92
```

### 分享卡內容

圖片卡可以是 1200x630，方便貼到社群平台。

包含：

- 遊戲 Logo
- 事故名稱
- 事故等級
- 主要截圖
- 玩家稱號
- 關鍵數據
- 安全委員會吐槽
- 挑戰碼

---

## 16. 挑戰碼設計

挑戰碼的作用是讓朋友能玩到同一場事故條件。

挑戰碼可以保存：

- 地圖 seed
- 玩家角色身份
- 場地機關配置
- 法術配置
- 起始位置
- 實驗艙位置
- 特殊規則

範例：

```text
MIR-7A92
```

或 URL：

```text
magic_incident_report.html?seed=7A92&mode=containment&lab=reactor
```

這會讓分享從「你去玩這遊戲」變成：

> 你來挑戰我這場事故。

---

## 17. 每日事故重現

原本的 Daily Challenge 可以改成：

> 今日事故重現

每天系統提供一個固定事故案例，讓所有玩家挑戰同一場。

範例：

```text
今日事故重現：第 44 號蒸氣爆炸事件
場地：元素反應室
條件：火 + 冰法術不穩定
目標：在收容程序啟動前存活 90 秒
```

這樣每日挑戰就不只是排行榜，而是世界觀的一部分。

## 18. 事故圖鑑

玩家可以收集各種事故紀錄。

| 編號 | 名稱 | 解鎖條件 |
|---|---|---|
| #001 | 火球誤爆事件 | 被自己的火球爆炸波及。 |
| #002 | 冰面滑倒事件 | 在冰面滑進危險區。 |
| #003 | 雷鏈反彈事件 | 雷鏈連到自己或隊友。 |
| #004 | 蒸氣壓力爆炸 | 火 + 冰造成範圍爆炸。 |
| #005 | 龍捲火災 | 風 + 火形成火焰旋風。 |
| #006 | 反向收容事件 | 收容員被關入實驗艙。 |
| #007 | 錯誤目標收容 | 抓錯目標。 |
| #008 | 同歸於盡 | 雙方同時死亡或同時被炸飛。 |
| #009 | 艙門短路事件 | 雷擊中斷收容倒數。 |
| #010 | 自行入艙事件 | 失控法師自己滑進或炸進艙。 |

事故圖鑑可以提供長期收集目標，不需要太多數值養成也能讓玩家想反覆遊玩。

## 19. 版本路線建議

### V0.8：事故報告雛形

先在現有魔法大亂鬥基礎上加入：

1. 結算事故報告。
2. 事故名稱生成。
3. 事故等級。
4. 安全委員會評語。
5. 事故稱號。
6. 複製分享文字。
7. 10 種事故圖鑑。

這一版不需要多人、不需要伺服器、不需要完整收容模式。

目標是先驗證：

> 事故報告這個包裝是否讓遊戲變得更有記憶點？

---

### V0.9：收容測試原型

加入簡化版 1v1 或 AI 對手模式。

核心功能：

1. 一名失控法師。
2. 一名收容員。
3. 實驗艙。
4. 穩定值 / 削弱值。
5. 推進實驗艙。
6. 3 秒關艙倒數。
7. 關艙失敗爆炸。
8. 結算事故報告支援收容事件。

可以先用本機 AI 測試，不急著做線上多人。

---

### V1.0：社群挑戰版

加入：

1. 挑戰碼。
2. 每日事故重現。
3. 事故分享卡。
4. 本機排行榜。
5. 最後 8 秒回放。
6. 自訂事故條件。

這會讓遊戲真正具備社群分享性。

---

### V1.1 之後：基地 / 地圖編輯器

延伸成更大的方向：

> 玩家可以設計自己的魔法實驗基地，讓其他玩家來逃脫或收容。

這可以接回之前討論過的：

- 家園經營
- 非同步 PvP
- 地圖編輯器
- 玩家自建防線
- 挑戰碼分享

但這是後期，不建議一開始就做。

## 20. 與原本魔法大亂鬥的關係

這不是完全推翻原本遊戲，而是把原本的戰鬥系統重新包裝成更有社群性的玩法。

### 原本保留

- Q 版 3D 魔法角色。
- 短局戰鬥。
- 火、冰、雷、風、土元素。
- 法術組合。
- 場地機關。
- 搞笑物理感。
- Boss / 怪物可作為中立干擾因素。

### 新增核心

- 收容目標。
- 實驗艙。
- 失控法師 vs 收容員。
- 魔法事故報告。
- 事故等級。
- 分享卡。
- 挑戰碼。
- 事故圖鑑。

---

## 21. 最大優勢

這個方向的最大優勢是：

> 失敗也會變好笑。

普通 PvP 輸了會不爽。  
但這種模式輸了可能會變成：

- 我本來要抓人，結果自己被關進艙。
- 我快逃出去了，結果自己放的冰把自己滑回實驗艙。
- 我們三個人追一個人，結果把基地炸爛還沒抓到。
- 我被自己的雷鏈反彈打斷逃跑。

這些都會變成玩家願意分享的故事。

## 22. 最終建議

目前最值得做的不是立刻增加大型內容，而是先確認新定位：

> 魔法事故報告 = 魔法實驗基地中的收容亂鬥 + 每局產生荒謬事故故事。

建議下一步先做：

1. 事故報告 UI。
2. 安全委員會評語。
3. 事故等級與事故名稱。
4. 簡單實驗艙物件。
5. 把敵人或 AI 角色推進艙的玩法測試。
6. 結束後產生可分享文字。

只要這個小原型好笑，就代表方向成立。

## 25. 總結

「魔法事故報告」可以補上原本魔法大亂鬥缺少的關鍵一塊：

> 社群分享性與戲劇性。

它讓遊戲從普通的打怪亂鬥，變成一款有強烈個性、可以產生搞笑故事、適合截圖分享的短局魔法收容遊戲。

最重要的是，這個方向不需要一開始就做成大型線上遊戲。  
可以先從單機原型開始：

- 一個實驗艙。
- 一個失控目標。
- 一個收容流程。
- 幾種場地事故。
- 一份搞笑事故報告。

只要玩家玩完會說：

> 這一局也太白癡了，我想再玩一次。

那這個方向就成立。
