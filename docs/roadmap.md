# 路線決策：打磨單機 vs 往下一階段

> 用途：記錄「現在這個單機 demo 接下來往哪走」的取捨與決策。
> 不重述玩法（見 [demo-concept.md](demo-concept.md)）與 BR 架構（見 [battle-royale-tech-stack-v1.md](battle-royale-tech-stack-v1.md)、[battle-royale-phase-plan-v1.md](battle-royale-phase-plan-v1.md)）。
> 狀態：**待決策**。記錄時間點：單機已做到 3D 體素版 v0.8（固定 45° 跟隨攝影機）。

---

## 0. 一句話

> 問題常被講成「**打磨單機** vs **做下一階段**」，但這是個假二選一。
> 真正的三條路是 A / B / C，而最關鍵的 C 常被藏起來。

---

## 1. 起點現況

- 單機玩法已實作並上線：四元素融合、元素反應、敵人波次、元素哥布林法師 Boss。
- 已從俯視 2D 演化到 **3D 體素 + 固定 45° 跟隨攝影機**（Three.js 渲染）。
- 手感已給朋友／陌生人玩過、回饋正面 —— 但**這是手感驗證，不是市場／留存數據**。
- 線上試玩：<https://pony0278.github.io/Mini-Mage-Mayhem/>

---

## 2. 三條路與取捨

### A. 繼續打磨現在的 3D／攝影機／美術
- **內容**：更多特效、更精緻模型、鏡頭/手感微調。
- **成本**：低。
- **回答了什麼決策**：幾乎沒有。不會告訴你「這遊戲值不值得做下去」。
- **關鍵提醒**：目前的 3D／體素／攝影機，**對 BR 完全用不到**（BR 伺服器是 headless 純邏輯，不含渲染）。
  所以「打磨單機畫面」**不等於**「往 BR 前進」。
- **槓桿**：三條裡**最低**（除非目的純粹是「上架前的賣相」，那歸到 C）。

### B. 直接跳去做 BR（即時多人大逃殺）
- **內容**：照 BR 文件做正式棧（Rust sim 核、dedicated server、Nakama、傳輸、場地同步）。
- **成本**：**以月計**、有伺服器、要連線、要營運。
- **擋路的未知（兩個會殺死專案的）**：
  1. 滿場「累積／擴散／連鎖」的元素混亂，在延遲下**同步得了嗎**？
  2. 人對人的魔法混亂對打，是**爽還是惱**？
- **正確做法（文件自己說的）**：別一路蓋完整設計，**先做兩根探針**驗掉上面兩點，過了再投正式棧。
- **注意**：探針第 0 步（B0）是把玩法抽成 headless `sim.js`，**也跟美術無關**。

### C. 把單機打磨到「能上架」，真的丟上 Poki / CrazyGames（近期建議）
- **內容**：收尾單機（開始/結算流程、音效、手機支援、平台 SDK、體積/載入…），實際上架。
- **成本**：低～中。
- **回答了什麼決策**：**最關鍵的那個**。BR 文件假設「單機已驗證通過」，但那是手感、不是
  **市場與留存數據**（玩家要不要這個循環、會不會回鍋、影片有沒有分享感）。
- **為何重要**：這份真實數據，才是 BR 的 Go/No-Go 真正該依據的東西。對應
  demo-concept.md 第 21、22 節的成功標準。

---

## 3. 決策建議

> **不要為了「畫面更美」而打磨；要為了「上架拿真實留存」而打磨。**

- **近期**：走 **C** — 把單機收尾上 Poki，用市場數據決定要不要投 BR。
- **若鐵了心要 BR**：別再碰美術（A），直接做 **B0 headless 抽取 + 探針一**（BR 階段計畫的階段 1）。
- **A 單純打磨**：只在「為了上架賣相」時才做，且歸進 C 的範圍內，不當成獨立目標。

### 一句話流程
```
現在 ──→ C（上架單機, 拿留存數據）──→ 數據好? 
                                         ├─ 是 → B（探針一/二 → GATE → BR 正式棧）
                                         └─ 否 → 調整單機設計 / 承認適合單機收手
```

---

## 4. 決策準則（什麼數據該推向哪條路）

- **C 之後該續做 BR（往 B）**，若：留存/回鍋率不錯、影片分享感成立、玩家明顯想要「跟別人一起玩」。
- **C 之後該收手或調整**，若：玩家玩一兩局就走、看不出重玩動機 → 先修單機核心循環，別把幾個月丟進 BR。
- **任何時候都別把「美術打磨」誤當成「專案進度」** —— 它不移動上面任何一個風險。

## 5. 下一步選項（待使用者拍板）

3. **暫不動工**：先讓本文件沉澱，想清楚方向再回來。
