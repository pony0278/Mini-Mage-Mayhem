# v2 規格 C — 最大災難回顧 / 一鍵分享（增長引擎）

> 對應：v2《環境處決 PvP》§0.5 C（會痛・增長引擎）、§8（結算）、§10。
> 一句話：**C 把 A/B 已產生的「有作者、好笑的死亡瞬間」自動做成一張帶標題與連結的截圖卡，一鍵分享。MVP 只做截圖卡（便宜），clip 之後再說。**
> 狀態：規格（增長引擎，非 fun 來源）。前置：規格 A（好笑）先過——對還不好笑的東西做分享系統 = 放大零。

---

## 1. C 是什麼、為什麼是增長引擎
A 把死亡做好笑、B 讓死法有作者——**C 是把「好笑那一局」轉成「可發布素材」的轉換器**。它直接把已發生的爆笑瞬間，做成玩家自願轉發的東西。沒有 C，笑聲留在房間；有 C，笑聲變成拉新人的影片。投報率可能比再加一個元素高。

## 2. 站在 A / B 的肩膀（C 大多是「收集 + 輸出」，不是從零做）
- **A 的致死鏡頭 beat**（拉近 + 凸眼臉在畫面中央）= 已經幫你**構圖**好。
- **B 的 `lastHitBy` 歸因** = 已經幫你寫好**標題**（「你把 X 轟下去了」/「X 自己掉進自己的局」）。
- 所以 A 的鏡頭、B 的歸因，**就是 C 的基礎建設**——現在做 A/B 時就把它們埋好。

## 3. 病毒學：PvP 分享強在哪
- **兩個傳播者**：加害者分享「我設計的死法」（ownership 炫耀）；受害者分享「我蠢死的瞬間」（self-deprecating，短影音超吃香）。一個瞬間 = 兩條傳播路徑。
- **有真人受害者** → 比單機「炸一堆 AI」更有戲、更有看好戲價值（最高分享性軸）。

## 4. MVP — 別一步到位做錄影
網頁現實：**截圖極便宜**（canvas → 圖檔）；**錄短片較重、易卡**（網頁免費玩家對 jank 零容忍）。分兩階：

| 階段 | 內容 | 成本 |
|---|---|---|
| **C1（先做）** | **災難回顧卡**：一局結束自動挑最戲劇性瞬間 → 一張**截圖 + 自動標題 + 浮水印/連結 + 存檔/分享鈕** | 低（截圖 + 既有歸因）|
| **C2（之後）** | 該瞬間做成 **3–5 秒短 GIF/clip**（引爆前後）| 中–高（錄製 + 效能）|

## 5. 怎麼挑「最戲劇性瞬間」
用**災難分數**啟發式（可搬 v1 的 `recordDisaster/makeRunStory`）：擊殺數、連環爆、把人轟下洞、被多人圍觀的死、自作自受的蠢死……分最高者即為卡片瞬間。每局自動記錄候選幀（A 的致死 beat 觸發時抓幀）。

## 6. 設計鎖（讓它真的被分享）
- **自動、不要求玩家操作**：結算畫面**直接**端上卡 + 分享鈕。要玩家自己去截圖 = 不會做。
- **構圖在 A 埋好**：致死瞬間鏡頭已對準受害者凸眼臉 → 直接抓那一幀。
- **標題自動生成**：`lastHitBy` + 死法 →「🔥 你把 PlayerX 轟進火海」/「💀 我自己滑進洞了」。一句話、好笑、可直接當貼文。
- **浮水印 + 連結**：卡片角落印遊戲名 + 一鍵進場連結 → 轉發即拉新。**這是病毒迴圈閉合的關鍵，常被忘記。**
- **低摩擦分享**：Web Share API（手機原生分享）/ 下載 / 複製連結。

## 7. 風險 / 紀律
- **別在 fun 驗證前做 C**：C 是放大器，不是 fun 來源。先 A（3–4 人笑出聲）確定有「值得分享的瞬間」，再做 C 輸出它。
- **C2 錄影 jank 風險**：MVP 先截圖，clip 等有資源再做。
- **不偏離**：C 服務「把已發生的好笑輸出」，不是「製造好笑」——別讓 C 搶走 A/B 的力氣。

## 8. 定稿
> **C = 自動把有作者的好笑死亡瞬間，做成帶標題與連結的截圖卡，一鍵分享。MVP 只做截圖卡；構圖與標題在 A（鏡頭）和 B（歸因）就已埋好。**
