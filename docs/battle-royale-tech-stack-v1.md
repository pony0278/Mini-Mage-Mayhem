# 魔法大逃殺 — 完整技術棧定案 (v1)

> 用途：把「即時動作魔法 roguelike → 即時多人大逃殺(BR)」這條路上收斂的技術選型固定下來。
> 回答：客戶端 / sim 核 / 即時對戰伺服器 / meta 後端各用什麼、界線在哪、什麼時候才升級。
> **核心前提**：單機版手感已給陌生人玩過、驗證通過；現在要做的是即時 PvP BR。
> **本文件只管技術棧與架構，不重述玩法設計。**

---

## 0. 一句話定位

一款 web 即時多人大逃殺。**伺服器權威**跑一顆 Rust 模擬核，客戶端是「預測 + 內插」的薄殼；
Nakama 只做 meta（配對 / 帳號 / 排行 / 社交），真正的即時對戰跑在**獨立的 Rust 專用伺服器**上。

> 注意：BR 會打破單機階段「Poki ≤8MB、無伺服器、AUDS 全包」那組約束。這是一個**有伺服器、要連線、以月計的多人專案**，不是「幫 roguelike 加一個模式」。

---

## 1. 最重要的一句話（整份文件的軸）

> **即時 PvP 和「滿場累積、會擴散、會連鎖的元素混亂」在 netcode 這條軸上直接打架。
> 你越成功把房間炸成一團，狀態就越難同步。你的招牌爽點正是最難連線的東西。**

整套架構的每個決定，都是在處理這句話。

---

## 2. 心智模型：哪些被「即時同步」綁死，哪些自由

1. **權威 sim 核** — 唯一一份真相。客戶端不自己算場地，只顯示伺服器送來的狀態。被「伺服器權威 + 即時同步」綁死。
2. **表現層（渲染 / 輸入 / juice / 預測）** — 不被綁死，可以演化。客戶端對自己角色做預測，對其他一切做內插。
3. **meta 後端（配對 / 帳號 / 排行 / 儲存）** — 跟即時同步無關，正常後端，Nakama 全包。

只有第 1 層是定海神針，其餘跟著它走。

---

## 3. 一個關鍵釐清：伺服器權威 ≠ 需要跨機 bit 級確定性

這點修正了「即時 PvP 一定要先做完整 determinism rewrite」的常見誤解：

- **lockstep / 非同步回放驗證**（沒有中央權威、大家各自算還要一致）→ 需要跨平台 bit 級確定性（定點數學、WASM/native 完全一致）。**這條很貴。**
- **本專案走的是伺服器權威 + 廣播快照** → 伺服器算一次就是真相，客戶端只顯示。**不需要**跨機 bit 一致；客戶端預測只要「夠接近、reconciliation 修正看不出來」即可。

因此：
- 伺服器端仍要 **固定步(fixed timestep) + 種子化 RNG**（為了穩定、可重現、好除錯）。
- 但**免掉**了最貴的那條 determinism 稅。這讓整個專案比想像中便宜一截。

## 4. 定案技術棧（總表）

| 層 | 用什麼 | 為什麼 | 界線 / 何時升級 |
|---|---|---|---|
| **sim 核** | **Rust → WASM(客戶端) + native(伺服器)** | 一份權威核同時餵客戶端預測與伺服器真相；預測與真相同源，reconciliation 修正量小，消掉一整類「兩邊算不一樣」的 bug | 不升級——定海神針 |
| **即時對戰伺服器** | **獨立的純 Rust dedicated server**（權威 tick） | 固定步、無 GC 停頓、跑混亂場地最穩；純 Rust 無 cgo，iterate/debug 單純 | 規模逼出問題前不動 |
| **客戶端 / 表現層** | **輕量 web**（現有 Canvas 演化；Phaser 視需要上） | web BR、吃低階手機；自己角色預測、其餘內插 | 要更多 juice/效能再上 Phaser/Pixi |
| **meta 後端** | **Nakama**（自架 Docker + Postgres） | 配對、帳號、排行、社交開箱即用；BR 的配對尤其關鍵 | 規模大需自訂時擴 runtime |
| **接法** | **B：解耦**（Nakama 管 meta，Rust GS 管對戰，token 交付） | 純 Rust 無 cgo、兩邊獨立崩潰邊界、配對與對戰各自擴縮 | 規模極小、求最少 moving parts 時才考慮 A(cgo 內嵌) |
| **fleet 營運** | **Agones 之類**（game-server 分配/擴縮） | BR 本來就需要按場分配專用伺服器 | 探針階段不碰 |
| **傳輸** | **WebSocket 起步** | TCP、可靠有序、一個 Node/ws 或 Rust ws server 就起得來 | 丟包下延遲成瓶頸 → WebRTC DataChannel(Netlib) |
| **場地狀態表示** | **粗格子 fuel/channel 網格**（取代圓形 zone 實體） | 既服務「殘留/疊加/重新引爆」，又是最好 delta 同步的網路表示 | — |

## 5. 各層細節

### 5.1 sim 核 — Rust（WASM + native）

- 介面：`step(state, inputs) → {state, snapshot, events}`，**無 DOM、無計時器、無 `Math.random`/`Date.now`**（種子外傳）。
- 客戶端用 `wasm-bindgen` 跑同一份做**預測**；伺服器跑 native 做**真相**。
- 固定步、tiebreak 寫死、任何影響狀態的迭代顯式排序（穩定、好除錯）。
- **不需要**定點數學/跨機 bit 一致（見 §3）。
- **現況**：sim 目前綁在瀏覽器 Canvas（v0.6.5，約 2300 行，含元素融合系統、四元素、反應、敵人）。**抽成 headless 是第一步（B0，見 §10）。**

### 5.2 即時對戰伺服器 — 純 Rust dedicated

- 每 match 一個（或一個 process 託管一房）權威 session。
- 每 tick 迴圈：收客戶端輸入 → 步進 sim → 產 snapshot → 廣播。
- **起手數值（拿來量、不是聖旨）**：權威 tick 20Hz 固定步；客戶端送輸入 30–60Hz（帶 seq# + dt）；快照廣播 20Hz；客戶端內插延遲 ~100ms。
- 無 GC 停頓是選 Rust 的硬理由（Go/JS 的 GC 在固定 tick 伺服器上會偶發抖動）。

### 5.3 客戶端 / 表現層 — 輕量 web

承重規則只有一條：

> **客戶端只預測「自己的角色」；對手、敵人、以及「場地」全部用伺服器快照內插。**

- **預測自己角色**：按鍵立即回饋；伺服器回「第 N 個輸入後的權威位置」，客戶端重跑未確認輸入做 reconciliation。
- **場地不預測、只內插**：哪裡有火、火傷不傷人，只有伺服器說了算 → 兩個玩家不可能對場地分歧。客戶端可加純視覺粒子（不影響傷害）。
- **為何吃得下這套延遲拆分**：場地是慢的、累積的，「你的動作影響場地要等約半個 RTT」可接受；要即時的走位是預測的。**快的東西預測、慢的東西交給伺服器**，本遊戲節奏正好落在對的一邊。

### 5.4 meta 後端 — Nakama

- 配對(matchmaker)、帳號(accounts)、排行、好友/大廳、儲存（Postgres）。
- 自帶權威 match runtime，但**本專案不用它跑對戰**（對戰交給獨立 Rust GS，見 §6）。
- 一句潑冷水：**Nakama 給你的是傳輸與生命週期，不是場地同步。** 你最難的「上百 zone 不 desync」還是你自己的，Nakama 不是銀彈。

## 6. 接法定案：B 解耦（不走 cgo 內嵌）

### 6.1 兩種接法比較

| 維度 | A：cgo 內嵌（Rust sim 連進 Nakama 單一 process） | B：解耦 dedicated（Nakama=meta，獨立 Rust GS=對戰）✅ |
|---|---|---|
| 部署 | 簡單：一個二進位、一次部署 | 多一個 game-server fleet 要營運 |
| 程式邊界 | cgo 縫：每 tick 跨 Go↔C↔Rust，FFI 開銷 + 建置複雜 | 純 Rust，無 cgo，iterate/debug 單純 |
| 崩潰半徑 | Rust panic 可能帶垮整個 Nakama（連配對/帳號） | 對戰掛了，配對/帳號還活著 |
| 擴縮 | 綁在一起 | 配對與對戰各自擴縮 |
| 適用 | 規模極小、求最少 moving parts | **會長成 BR 的東西（本專案）** |

**定案：走 B。** 若真要走 A，FFI 邊界務必**粗顆粒**（每 tick 丟整包 state、拿整包 snapshot+events，絕不 per-entity 呼叫）。

### 6.2 交付握手（Client ← Nakama → Rust GS，token 驗證）

信任根：**token 由 Nakama 簽、Rust GS 驗**，把信任從 meta 層傳遞到對戰層。

1. **Client → Nakama**：登入 + 請求配對。
2. **Nakama（self）**：配對湊滿人 → 向 fleet 分配器(Agones)要一台 Rust GS 拿位址 → 簽發 token(JWT)，內含 `matchId`、`playerId`、允許玩家清單、**短到期**(如 30s)、**一次性 nonce**；用 Nakama↔GS 的**共享信任根**簽名（共享 HMAC secret，或 Nakama 私鑰簽 / GS 公鑰驗）。
3. **Nakama → Client**：回傳 `{ GS 位址, token }`。
4. **Client → Rust GS**：連線，第一個訊息帶 token。
5. **Rust GS（self）驗 token**（純本地、不問 DB、無狀態）：
   - **簽章**：確認真的是 Nakama 發的。
   - **到期**：擋掉舊 token。
   - **scope**：`matchId` 是這台分配到的那場、`playerId` 在允許清單 → 擋「拿 A 場票闖 B 場」。
   - **防重放**：nonce 用過即作廢、綁定此連線。
6. **Rust GS → Client**：全過 → admit 進 session，開始 input/snapshot 迴圈；任一項失敗 → 關連線。

好處：GS **無狀態驗證、不需回頭問 Nakama**（簽章自帶信任）→ 快、可水平擴展；**沒有 Nakama 簽的票，沒人能連進對戰伺服器** = 伺服器權威防線的入口鎖。

---

## 7. 傳輸

- **起步：WebSocket（TCP）。** 可靠有序、簡單、2 人小場綽綽有餘。
- **升級：WebRTC DataChannel（即你舊文件的 Netlib）。** 價值是「丟包下的低延遲」，上線後才需要。
- **換傳輸不動架構**（伺服器權威模型不變），所以現在延後是安全的。

---

## 8. 場地狀態層（殘留 / 疊加 / 重新引爆）+ 網路同步的雙重理由

- 設計性質：燃料(fuel)會**累積、殘留**；點火源碰到累積燃料就連鎖**重新引爆**。
- **資料模型**：每格帶多個獨立純量通道（`fuel`/`fire`/`wet`/`poison`/`charge`），而非單一 enum（單 enum 做不出疊加）。
- **更新模型**：固定頻率 CA tick（~10Hz，與渲染解耦）、雙緩衝（避免擴散方向性偏差）。30×30 全掃即可，先別做 active-set。
- **網路理由（新增）**：粗格子 delta 起來又小又乾淨（只送變了的格），是最好同步的場地表示；現行「一堆圓形 zone」難 delta、難同步。
- **結論**：「zone → fuel 網格」這個遷移現在有**兩個**理由（場地累積 + 網路效率），是同一個重構。探針階段可先用現有 zone 硬送、**先量頻寬數字**。

---

## 9. BR 專屬系統（在「即時 PvP」之上再疊一層）

- **縮圈**：不然拖戲。
- **局內進度**：地圖撿經驗 → 升級 → 隨機抽法術卡；擊殺給經驗。
- **滾雪球追趕設計**：早殺→早升級→更強的雪球在 BR 會造成「領先者輾壓、其他人體驗差」，**必須設計追趕機制**（落後補償、後段經驗加成等）。
- **人口冷啟動**：BR 沒有足夠同時在線就湊不滿房間，空房=遊戲死；早期通常**用 bot 填房**。
- **觀戰 / 快速重排**：死了要能觀戰或秒重排，降挫折。

---

## 10. 反作弊立場

- Poki 探針階段「先驗好不好玩、不驗平不平衡」→ 可接受。
- 正式 BR：有進度的競技一定有人作弊，**伺服器權威是唯一防線**（客戶端權威=秒破：加速/無限經驗/透視）。
- token 的 scope + 到期 + 防重放（§6.2）是入口鎖；對戰中所有判定由伺服器 sim 拍板。

---

## 11. 開發順序：探針優先，通過才上正式棧

整套 BR 蓋下去要數月。**不要照完整設計一路做**，先用最小成本驗掉兩個會讓專案死掉的未知：

### 探針一（技術生死題）：netcode 可行性
- 2 玩家、一個小競技場、共享一片火/毒場地、伺服器權威、客戶端預測。
- 目標：延遲下兩人看到**同一片場地**、打起來手感可接受。
- **量三個數**：場地是否一致？延遲下手感如何？RTT 與頻寬多少？

### 探針二（設計生死題）：競技到底好不好玩
- 2–4 人 FFA 小房，無縮圈、無觀戰、固定法術。
- 只驗：人對人的魔法混亂對打是爽還是惱？場地連鎖在對抗下成立還是變暴怒源？

> **兩根探針都過，才蓋上面的 BR 上層建築（多人、縮圈、抽卡升級、配對、觀戰、進度）。**
> 更輕的中間路線：直接做「4–8 人即時 FFA 競技場」當產品先上，省掉縮圈/配對/觀戰一大堆周邊，早幾個月驗證「玩家要不要這個多人」，再長成 BR。

### B0：探針的第 0 步 — 抽 headless sim
- 把 v0.6.5 已驗證好玩的玩法，抽成 `step(state, inputs) → {state, snapshot, events}` 的純函式模組，無 DOM、種子外傳。
- 它一份同時餵：客戶端(WASM/或先 JS)、伺服器(native/或先 Node)。
- **無論後端怎麼決定都不變、且現在就能動**的第一步。

## 12. 探針一的最小規格（重要：別用正式棧蓋探針）

> 探針的任務跟「用什麼語言/後端」**無關**。別為了探針就 port Rust + 架 Nakama + 接 cgo——那等於在「專案活不活得了」還沒回答前，先把產品成本大半付掉。

| 項目 | 探針一用什麼 | 為什麼 |
|---|---|---|
| 對戰伺服器 | **Node/TS 權威 server**（跑現有 JS sim 的 headless 版） | JS sim 已在 → 幾天就能動 |
| 傳輸 | 純 **WebSocket** | 最簡單，2 人夠用 |
| 房間 | **寫死一個房間、2 玩家、1 張小競技場** | 砍掉配對/帳號/縮圈 |
| 內容 | 移動 + 一個法術 + 共享一片場地 | 只驗同步與手感 |
| Nakama | **不碰** | Nakama 是 meta 層，跟「場地同步得了嗎」無關 |

學到的 netcode 知識（快照格式、delta 策略、內插調校、頻寬數字）**換不換語言都帶得走**。

（若已鐵了心走 Rust、JS 反正要丟：探針可用「最小 Rust server + 純 WebSocket」順便測 Rust 合不合手——**但一樣別碰 Nakama**。）

## 13. 升級觸發條件（什麼時候才做下一步）

| 升級 | 觸發條件 |
|---|---|
| JS sim → Rust sim 核 | 探針驗證通過、決定認真做 BR |
| 探針 server → 正式 Rust dedicated + Nakama(B 接法) | 同上 |
| WebSocket → WebRTC(Netlib) | 丟包下延遲成為實測瓶頸 |
| 手動部署 → Agones fleet | 需要按場分配/擴縮專用伺服器 |
| zone 實體 → fuel 網格 | 場地累積玩法定案 或 場地同步頻寬成瓶頸 |
| 無 bot → bot 填房 | 同時在線不足以湊滿房間 |

## 14. 風險清單

1. **🔴 場地同步是專案成敗的核心未知**：上百 zone 在延遲下不 desync —— 這是探針一要驗的命根，沒過 BR 不可能。
2. **🔴 設計風險**：PvE 迷人的「累積打法 + 自傷搞笑」在 PvP 可能變「龜縮封鎖 + 被延遲炸死的暴怒」，且執行門檻把受眾往硬核拉。需探針二驗。
3. **滾雪球**：經驗領先者輾壓全場，需追趕機制。
4. **人口冷啟動**：空房=遊戲死，早期需 bot。
5. **範圍蔓延**：這是月計的多人後端專案，不是加一個模式——心理與資源都要照這個量級準備。
6. **fleet 營運成本**：B 接法多一塊 game-server 分配/擴縮（Agones 等）。
