## 階段 1 — 探針一：netcode 可行性 ★現在做

**目標**：兩個人在延遲下看到**同一片會動的世界**。回答「混亂場地同步得了嗎」。
**平台**：全程 localhost 兩個瀏覽器分頁，先不上雲。

- [ ] **1.0 環境**：開資料夾、`npm init -y`、`npm i ws`（唯一依賴）。 — 約 10 分
- [ ] **1.1 抽 headless sim（B0，最關鍵、最易低估）**
  - 從 v0.6.5 抽出純邏輯成 `sim.js`：實體、碰撞、場地反應。
  - 拿掉所有 DOM / Canvas / `requestAnimationFrame` / 鍵鼠監聽。
  - 對外只留 `step(state, inputs, dt) → state`。
  - `Math.random` → 帶種子 PRNG（如 `mulberry32`），種子由 server 給。
  - **完成標準**：`node` 裡建 state、跑 100 tick 不崩、玩家座標會變。 — 約 1～2 天
- [ ] **1.2 server 骨架**：`server.js` 起 `ws`、**寫死一房**（一份 state + `[socketA, socketB]`），前兩個連入當 player 0/1。 — 約 半天
- [ ] **1.3 固定步 tick**：`setInterval` 20Hz → 餵兩人輸入進 `step()` → 推進 state → 廣播 snapshot。**dt 用 server 固定 1/20，不用 client 時間。** — 約 半天
- [ ] **1.4 client 改造（先求一致）**：拔掉本地 sim；每幀送輸入；收 snapshot 直接畫；**自己角色也先等 server 回來才動（會頓，故意的）**。
  - **完成標準**：兩個分頁看到同一個世界、彼此會動。**← 探針最大的問題在這一刻回答。** — 約 1 天
- [ ] **1.5 預測 + 內插（讓它好玩）**：自己角色本地預測 + reconciliation；其他一切 buffer 2~3 個 snapshot 內插（落後 ~100ms）。
  - **完成標準**：自己即時、別人平順。 — 約 2～3 天
- [ ] **1.6 量數字**：debug 角落顯示 RTT、每秒 snapshot bytes。 — 約 半天

**階段 1 產出**：一個能跑的最小權威 server + 改造過的 client，以及三個數字——**一致性 / 延遲手感 / 頻寬**。

**新手踩雷預防**
- 先 localhost、先傳 JSON（別碰二進位/delta）。
- **鐵律**：client 只送 input、只畫 snapshot，**絕不自己改 game state**。
- 卡在 1.1 是正常的（sim 跟畫面糾纏），回來討論。

## 階段 2 — 探針二：PvP 好不好玩 ★現在做

**目標**：回答「人對人的魔法混亂對打，是爽還是惱」。
**做法**：在階段 1 的骨架上，開 2～4 人 FFA 小房，**無縮圈、無觀戰、固定法術**。

- [ ] **2.1 把房間人數從 2 開到 4**（FFA）。
- [ ] **2.2 加最小勝負**：剩一人 = 贏；死了能快速重排一局。
- [ ] **2.3 固定一套法術 + 場地反應**（先別做抽卡升級）。
- [ ] **2.4 找幾個真人對打、觀察**：
  - 場地連鎖在對抗下是爽點還是暴怒源？
  - 累積打法變成龜縮封鎖了嗎？
  - 「被對手/被延遲炸死」的挫折可接受嗎？

**階段 2 產出**：對「這個玩法搬到 PvP 還成不成立」的真人手感判斷。

## ◆ GATE — Go / No-Go 決策

**用兩根探針的數據與手感，決定要不要投正式棧。** 這是省幾個月的那道閘門。

**Go（往下做）需要大致滿足：**
- [ ] 兩個 client 在延遲下看到的場地**一致、不 desync**。
- [ ] 頻寬在可接受範圍（且看得出能撐到多人）。
- [ ] 延遲下的施法/閃避手感**可接受**。
- [ ] 真人 PvP 對打**偏爽而非偏惱**。

**No-Go（先別投）：**
- 同步做不出來 → 回頭：改場地表示（zone → fuel 網格）、降同步量，或重估玩法。
- PvP 不好玩 → 回頭調設計，或承認這款適合單機/合作、收手不做 BR。

> No-Go 不是失敗，是**用最小成本買到「不要把幾個月丟進水裡」的資訊**。
